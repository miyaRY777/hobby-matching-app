# 最重要ルール（絶対遵守）

- 設計合意前に実装を始めない
- RED -> GREEN -> REFACTOR を基本とする
- すべての確認系コマンドは `docker compose exec web` 経由を優先する
- 設計合意後に Issue と作業ブランチを用意してから実装を始める
- 完了報告前に、実行したテストや検証結果を明示する
- PR 前に RSpec / RuboCop を通す
- 不明点や仕様の分岐がある場合は、推測で埋めずに停止して確認する
- 仕様に曖昧さが残る場合は実装せず Phase 0 に戻る

---

# Codex 作業フロー

## Phase 0: インテイク

- 依頼内容を要約する
- 目的、スコープ、受入条件、制約を整理する
- 未確定事項が大きい場合は、実装前に確認する
- 未確定事項が残る限り、実装提案や実装開始をしない

## Phase 1: 設計合意

- データ構造、DB制約、責務分離、N+1、トランザクション要否を確認する
- 複数案があるときは、差分とトレードオフを明示する
- 設計合意後に Issue と作業ブランチを作成する
- GitHub Project「Hobby Matching」で管理している作業は、Issue とブランチの対応を意識する

## Phase 2: 実装計画

- 変更対象ファイルを洗い出す
- 追加・更新するテストを決める
- RED / GREEN / REFACTOR の順に進める
- 必要なら Service 分離や責務移動の要否を確認する
- この計画で進めてよいかを確認してから実装する

## Phase 3: 実行

- 先に失敗するテストや再現手順を用意する
- 最小変更で通す
- 通ったあとに責務や命名を整える
- 最後に受入条件と実行結果を突き合わせる
- テスト失敗やバグ調査では、推測で直さず根本原因を確認してから修正する
- 修正方針に分岐がある場合は、その場で止まって確認する
- レビュー用のサブエージェント相当は、ユーザーが明示的に望んだ場合にのみ使う

# TDDルール

- 実装を先に書かない
- 挙動変更やバグ修正では、再現テストを先に書く
- `it` は 1 関心ごとを基本とする
- テストが通るだけでなく、要件を満たしているか確認する

## 省略できるケース

- 文言、コメント、見た目だけの軽微修正
- ロジック、DB、認可、例外処理に影響しない変更
- ただし迷ったらテストを書く

---

# コミット粒度ルール

- 責務単位でコミットを分ける
- 1行で説明しにくい変更は分割を検討する
- コミット前に、何をどこで区切るかを意識して整理する
- 大きい変更は、コミットの分け方を先に言語化してから進める
- コミットメッセージは `fix:` `feat:` `refactor:` `docs:` などの prefix は英語でそろえ、説明は日本語で書く

---

# 行動原則

## 検証なくして完了を主張しない

- 推測で「たぶん通る」と言わない
- テスト通過と受入条件の達成を分けて確認する
- 過去の結果を使い回さず、今回の実行結果で判断する

## 根本原因を特定してから修正する

- バグに遭遇したら、先に再現・観察・原因特定を行う
- 1回に1つの仮説だけを検証する
- 3回連続で修正が外れたら停止して相談する

## 危険信号

| 思考 | やるべきこと |
|---|---|
| 「とりあえず直してみよう」 | 先に再現手順と原因を確認する |
| 「たぶん大丈夫」 | テスト・確認コマンドで証拠を取る |
| 「もう1回だけ試そう」（3回目） | 停止して方針確認する |
| 「テストは後で書こう」 | 先に RED を作る |

# 完了条件

- 変更内容を自分の言葉で説明できる
- 関連テスト、または必要な検証を実行済みである
- 既知の未対応や残リスクがあれば明示する

---

# 禁止事項

- Phase を飛ばして進める
- テストなしでロジック変更を入れる
- RuboCop 未確認のまま完了扱いにする
- 確認前に次の大きなステップへ進む
- Controller を安易に肥大化させる
- DB 制約で担保すべきことをアプリ層だけで済ませる
