## 最重要ルール（絶対遵守）

* **Issueを作成してから**作業開始（GitHub Project「hobby-matching-app」に作成）
* ブランチ作成 → **RED → GREEN → REFACTOR** → PR
* すべてのコマンドは `docker compose exec web` 経由
* PR前に **RSpec / RuboCop 全通過**
* 不明点が1つでもある場合、必ず質問する
* 推測で実装しない
* 不明点が1つでもある場合、必ず質問する
+ * 仕様に曖昧さがある場合、実装せずPhase 0に戻る

---

# 開発フロー（強制フェーズ制）

## Phase 0：インテイク（最初のターン）

実装に入らない。

1. 依頼内容を1〜3行で要約
2. 目的 / スコープ / 受入条件 / 制約を整理する
3. 未確定事項を列挙し、質問する

⚠ 未確定が残る限り、実装提案は禁止

---

## Phase 1：設計合意

* データ構造変更・既存モデル影響・DB制約・N+1・トランザクションを明確化する
* 複数案がある場合は 案A / 案B / 推奨案 / トレードオフ を提示する

合意が取れるまで次へ進まない。

---

## Phase 2：実装計画（Plan）

* 変更対象ファイル一覧
* 追加/修正テスト一覧
* RED → GREEN → REFACTOR の分解
* Service分離の要否

「この計画で進めてよいか？」を必ず確認。

---

## Phase 3：実行

* RED → GREEN → REFACTORを厳守
* 途中で方針が分岐したら停止して確認
* 高リスク操作前は必ず確認
* 各主要ステップで「何をしたか / 影響範囲 / 次にやること」を報告する

---

# TDD強制ルール

* 実装を先に書かない
* 修正時は必ず再現テストを書く
* itは「1関心ごと1例」
* テストが通っただけでは完了ではない（REFACTOR必須）

## 例外（TDDを省略できる条件）

次の **すべて** を満たす場合のみ省略してよい：

* 1ファイル内の軽微な修正（typo・文言・UIクラス・コメント）
* ロジック・DB・バリデーション・認可/認証・例外処理に **影響しない**
* 「なぜ不要か」をIssueに記録できる

省略する場合でも `rspec` と `rubocop` は必ず実行する。迷ったら **TDDを行う**。

---

# 禁止事項

* Issueなしで実装開始
* テストなし実装
* 推測による修正
* Controller肥大化
* DB制約を無視
* RuboCop未解消のままPR
* 確認なしに `git commit` / `git push` を実行する
* 確認なしにファイルを削除する
* 大量のファイルを一度に変更する（影響範囲を確認し段階的に進める）
* Phase をスキップする（Phase 0 → 1 → 2 → 3 の順序を守る）
